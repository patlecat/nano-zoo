# Glossar: SDLC-Begriffe fuer Kids

Dieses Glossar erklaert schwierige Begriffe aus den Skill-Dateien fuer den professionellen SDLC-Workflow in einfacher Sprache.
SDLC bedeutet: der ganze Weg von einer Idee bis zu getesteter Software.

## Begriffe von A bis Z

- **Abhaengigkeit**  
  Etwas, das zuerst fertig sein muss, bevor du weitermachen kannst.

- **Akzeptanzkriterium (AK)**  
  Eine klare Bedingung, die zeigt: "Ja, die Aufgabe ist richtig erledigt."

- **Approved**  
  Bedeutet: freigegeben, passt so.

- **Artefakt**  
  Ein Arbeits-Ergebnis als Datei, zum Beispiel ein PRD, ein Review oder eine Story-Datei.

- **Backend**  
  Der "unsichtbare" Teil einer App im Hintergrund, der Daten und Logik verarbeitet.

- **Blocker**  
  Ein Problem, das dich stoppt, bis es geloest ist.

- **Codebasis**  
  Alle Dateien mit Programmcode in einem Projekt.

- **Deployment**  
  Der Schritt, bei dem neue Software fuer echte Nutzer veroeffentlicht wird.

- **E2E-Test (End-to-End-Test)**  
  Ein Test, der einen ganzen Ablauf von Anfang bis Ende prueft, wie ein echter Nutzer.

- **Evaluation**  
  Eine Bewertung: Was ist gut, was fehlt, was muss besser werden?

- **Feature**  
  Eine Funktion der App, also etwas, das sie kann.

- **Findings**  
  Gefundene Auffaelligkeiten im Review, zum Beispiel Fehler, Risiken oder Unklarheiten.

- **Flow / User-Flow**  
  Die Reihenfolge von Schritten, die ein Nutzer in der App durchgeht.

- **Frontend**  
  Der sichtbare Teil einer App: Buttons, Seiten, Texte, Bilder.

- **Frontmatter (YAML-Frontmatter)**  
  Ein Info-Block ganz oben in einer Markdown-Datei mit Metadaten wie `artifact_type` und `Status`.

- **Handoff**  
  Eine geordnete Uebergabe an den naechsten Schritt oder Skill.

- **Implementierung**  
  Das Umsetzen einer Idee in echten Code.

- **In/Out (Scope In/Scope Out)**  
  "In" bedeutet: gehoert zur Aufgabe. "Out" bedeutet: gehoert bewusst nicht dazu.

- **Inkrement**  
  Ein kleines, nutzbares Zwischen-Ergebnis, das bereits Wert hat.

- **Integration**  
  Das Zusammenspiel mehrerer Teile, damit alles zusammen funktioniert.

- **Machbarkeit**  
  Die Frage: Ist das mit Zeit, Wissen und Technik realistisch umsetzbar?

- **Metrik / Erfolgsmetrik**  
  Eine messbare Zahl, mit der man sieht, ob ein Ziel erreicht wurde.

- **Needs Revision**  
  Bedeutet: muss ueberarbeitet werden, noch nicht gut genug.

- **Nicht-funktionale Anforderung**  
  Regeln zur Qualitaet, z. B. Geschwindigkeit, Sicherheit oder Zuverlaessigkeit.

- **Orchestrator**  
  Ein "Verkehrsleiter", der entscheidet, welcher Workflow-Schritt als naechstes kommt.

- **Persona**  
  Eine erfundene Beispiel-Person aus der Zielgruppe, damit man besser fuer Nutzer plant.

- **PRD (Product Requirements Document)**  
  Ein Dokument, das erklaert, was gebaut werden soll und warum.

- **Regression**  
  Ein alter Fehler oder ein neues Problem, das nach einer Aenderung ploetzlich auftritt.

- **Review**  
  Eine strukturierte Pruefung eines Dokuments oder Codes.

- **Risiko**  
  Etwas, das spaeter zu einem Problem werden kann.

- **Routing-Entscheidung**  
  Die Entscheidung, welcher Skill oder Schritt als naechstes ausgefuehrt wird.

- **SDLC (Software Development Life Cycle)**  
  Der komplette Lebenszyklus von Software: Idee, Planung, Umsetzung, Test, Verbesserung.

- **Scorecard**  
  Eine Bewertungstabelle mit Punkten, damit Entscheidungen nachvollziehbar sind.

- **Scope**  
  Der genaue Umfang einer Aufgabe: was drin ist und was nicht.

- **Split-Plan**  
  Ein Plan, eine zu grosse Aufgabe in kleinere, machbare Teile aufzuteilen.

- **Stakeholder**  
  Menschen, die vom Projekt betroffen sind oder wichtige Entscheidungen treffen.

- **Status-Lifecycle**  
  Die feste Reihenfolge von Statuswerten, z. B. `Entwurf` -> `In Bearbeitung` -> `Bereit zur Abnahme` -> `Erledigt`.

- **Story Point / T-Shirt-Groesse**  
  Schaetzung, wie gross oder aufwendig eine Aufgabe ist.

- **Testbarkeit**  
  Wie gut man pruefen kann, ob etwas wirklich korrekt funktioniert.

- **Traceability (Nachverfolgbarkeit)**  
  Man kann klar sehen, welches Akzeptanzkriterium durch welchen Code und welchen Test erfuellt wird.

- **UX (User Experience)**  
  Das Erlebnis eines Nutzers: Ist die App klar, angenehm und einfach zu benutzen?

- **Validierung**  
  Pruefung, ob Eingaben und Dateien den Regeln entsprechen.

- **Vitest**  
  Ein Test-Werkzeug fuer JavaScript/TypeScript-Projekte.

- **Workflow**  
  Eine feste Reihenfolge von Arbeitsschritten.
